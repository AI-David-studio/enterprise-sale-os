# Этап 14: UX Polish & Product Quality Hardening

## 1. Название этапа
Stage 14: UX Polish / Product Quality Hardening (Улучшение пользовательского опыта и отладка качества существующих интерфейсов)

## 2. Назначение
Устранение шероховатостей пользовательского опыта (UX) в существующих интерфейсах продавца (Этапы 9–13). Этап фокусируется на консистентности форм, обратной связи при успешных/ошибочных действиях и защите от случайных ошибок при вводе данных. Никакой новой бизнес-логики, изменения базы данных или новых функций не вводится.

## 3. Почему Этап 14 идёт после Этапа 13
Этапы 9-13 заложили полный фундамент MVP-функционала: воронка, задачи, коммуникации, документы, ИИ-интеграция и сводный дашборд. Все эти модули функционально работают и используют React 19 `useActionState`, однако визуальная обратная связь (уведомления об успехе, валидация полей до отправки, состояние загрузки) реализована базово и фрагментарно. Прежде чем переходить к сложным фичам (клиентский портал, аналитика), необходимо стабилизировать и отполировать уже созданную базу для продавца.

## 4. Аудит текущего UX-покрытия

### Real and already solid (Работает хорошо)
- Защита маршрутов через middleware и `getCurrentUserActiveDealContext`
- Базовое использование `useActionState` во всех мутациях
- RLS Этапа 8 корректно блокирует несанкционированный доступ

### Real but rough (Работает, но требует полировки)
- **Модальные окна форм** (`CreateBuyerForm`, `CreateTaskForm` и др.): Закрываются при успехе, но пользователь не получает глобального визуального подтверждения (toast-уведомления), что усложняет понимание результата, особенно если созданный элемент появляется за пределами видимости экрана.
- **Инлайн-ошибки**: Формы возвращают общую ошибку (`state.error`), но не подсвечивают конкретные поля.
- **Инлайн-мутации** (`TaskStatusToggle`, `ReviewButtons`, `PipelineStageSelector`): Ошибки выводятся мелким текстом рядом с кнопкой, а успех часто не сопровождается визуальным подтверждением, кроме изменения самих данных (что не всегда очевидно мгновенно).
- **Сброс форм**: Состояние полей модального окна не всегда надежно сбрасывается при повторном открытии.

## 5. Аудит качества рабочих процессов (Workflows)

| Процесс | Текущее состояние | Зона роста UX |
|---|---|---|
| **Deal settings update** | ✅ Текстовый success inline | Форма длинная; нужен глобальный toast |
| **Buyer creation** | ✅ Модалка закрывается при успехе | Нет глобального toast; ошибки не привязаны к полям |
| **Pipeline movement** | ✅ Inline text success | Успех исчезает только после перезагрузки компонента |
| **Task / Comm / Doc creation** | ✅ Модалка закрывается при успехе | Нет toast; форма может сохранять старые данные при быстром переоткрытии |
| **Task status toggle** | ✅ Inline error | Нет индикации успеха; иконка меняется, но нет текстового подтверждения в случае сложной страницы |
| **AI review approve/discard** | ✅ Inline error | Аналогично task toggle |
| **Dashboard** | ✅ Read-only, solid | Практически не требует полировки (уже отлажен в Этапе 13) |

## 6. Возможные варианты охвата Этапа 14

- *Вариант А: Глобальная консистентность форм и обратной связи (Выбран).* Добавление системы глобальных toast-уведомлений, стандартизация обработки ошибок в формах, улучшение состояний загрузки (isPending).
- *Вариант Б: Визуальный редизайн.* Замена сырых Tailwind-компонентов на UI-библиотеку (например, shadcn/ui). (Отклонён: слишком большой объем изменений, высокий риск поломки существующих рабочих процессов).
- *Вариант В: Адаптивность и мобильная верстка.* Глубокая переработка layout для мобильных телефонов. (Отклонён: MVP ориентирован в первую очередь на веб/десктоп, мобильная версия вторична на данном этапе).

## 7. Выбранный охват Этапа 14 (Locked Scope)

**UX Polish: Формирование единого стандарта обратной связи и стабильности форм.**

**Что будет улучшено:**
1. **Глобальные Toast-уведомления (при необходимости)**: Проверить репозиторий на наличие существующей системы уведомлений. Если она есть — использовать её. Если нет — допускается установка ровно одной легковесной библиотеки (например, `react-hot-toast`, `sonner` или аналогичной), локальной для Этапа 14. Не зацикливаться на конкретном пакете на этапе планирования.
2. **Стандартизация модальных форм (Очистка и Сброс)**: Соблюдение строгого контракта:
   - Форма очищается **только после успешной** отправки (`state.success === true`).
   - Форма очищается при преднамеренном закрытии/отмене пользователем.
   - Форма **НЕ очищается** при неудачной отправке (ошибке сервера/валидации).
   - Форма **НЕ очищается** в процессе отправки (`isPending === true`).
   - Повторно открытое модальное окно не должно содержать `state.error` или `state.success` от предыдущего состояния.
3. **Сохранение инлайн-ошибок для полноформатных форм**: Для больших форм (`DealSettingsForm`, `CreateBuyerForm`, `CreateTaskForm`, `UploadDocumentForm`, `CreateCommunicationForm`) инлайн-текст ошибки **сохраняется**. Запрещено слепо заменять все ошибки на всплывающие toast. Инлайн-ошибка часто удобнее для пользователя. Toast используется в них преимущественно для `success`.
4. **Улучшение инлайн-мутаций (Micro-actions)**: Для мелких кнопок в таблицах/списках (`TaskStatusToggle`, `ReviewButtons`, `PipelineStageSelector`), где инлайн-текст ломает верстку или вызывает скачки (layout shift), локальный текст ошибки/успеха заменяется на вызов `toast.error` / `toast.success`.
5. **Запрет двойного сабмита**: Строгая блокировка кнопок (disabled) во всех формах при `isPending = true`.

## 11. Ролевая модель и авторизация

- Этап 14 **НЕ** меняет текущих политик доступа.
- Если кнопка или форма в скрыта от роли `viewer` (как это реализовано сейчас) — она остается скрытой в Этапе 14.
- Визуально неактивные элементы для ролей с ограниченными правами (например `disabled` кнопки в Deal Settings для не-lead_advisor) сохраняются "как есть".

## 12. Целевое поведение

Пользователь (Продавец) нажимает "Добавить задачу". Вводит данные, нажимает "Создать". Кнопка блокируется, текст меняется на "Создание...". После успешного ответа сервера модальное окно закрывается, и в углу экрана (или сверху) всплывает неблокирующее зеленое уведомление: "Задача добавлена". Форма возвращается в пустое исходное состояние для следующего раза. В случае ошибки — выводится красное уведомление об ошибке.

## 13. Границы исполнения (Execution Boundaries)

- **Разрешенные изменения**: Установка 1 npm-пакета (напр. `react-hot-toast`), редактирование клиентских файлов компонентов (формы, кнопки), добавление `<Toaster />` в базовый layout `(protected)`. Рефакторинг клиентских хуков внутри `.tsx` файлов для корректной интеграции toast.
- **Запрещенные изменения**: Редактирование файлов `supabase/migrations/*.sql`, удаление `useActionState`, редактирование `src/actions/*.ts`, изменение логики маршрутизации, любые изменения логики `getCurrentUserActiveDealContext`.

## 14. Критерии приемки

1. При необходимости установки Toast-пакета — провайдер внедрен в `layout.tsx` (или использован существующий).
2. Микро-экшены (`Status Toggle`, `Pipeline Selector`, `AI Review`) не ломают верстку и используют toast для успеха/ошибки.
3. Полноформатные модальные окна и настройки сохраняют inline-ошибки, но добавляют системные уведомления для подтверждения успеха.
4. Контракт сброса форм строго выполнен: поля очищаются/резетаются только при успехе или закрытии, но никогда при ошибке или `isPending`.
5. Кнопки `Submit` во всех формах из scope блокируются (disabled/pending состояние) во время серверного вызова.
6. Выполнено строго в разрешенных файлах. Файлы в `src/actions/*.ts` не изменились.
7. Миграции и механизмы БД не изменились. Проект собирается без ошибок (`npm run build`).

## 15. Условия остановки (Stop Conditions)

- Исполнитель пытается изменить серверные контракты (например, начать возвращать полные объекты данных вместо boolean флагов из Actions) для реализации UX.
- Исполнитель добавляет обертки безопасности на клиенте, дублирующие серверные (Stage 8 RLS не должен дублироваться/усложняться клиентской логикой скрытия данных).
- Исполнитель меняет структуру базы данных, RLS или миграции.
