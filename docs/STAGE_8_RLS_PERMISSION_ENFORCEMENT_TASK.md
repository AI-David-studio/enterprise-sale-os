# Этап 8: Обеспечение прав доступа на уровне RLS (Финальная спецификация)

## 1. Цель этапа
Данный документ определяет строгий и безопасный для выполнения (execution-safe) план Этапа 8: перенос прав доступа на базу данных (Row Level Security). Этап отменяет обходной путь "организационного моста" (org-bridge) и превращает членство в сделке (`deal_members`) в единственный истинный источник контроля над доступом (source of truth) для таблиц сделки.

## 2. Зафиксированные концепции и границы (Базовые условия)
- **Сначала продавец / Сначала одна сделка**.
- **Доступ строго по приглашениям**: Совместная работа возможна только через приглашения (нет портала покупателей).
- **Роли**: Только `lead_advisor`, `advisor`, и `viewer`.
- **Identity != Role**: `public.users.user_type` — это тип учетной записи (например, 'seller' или 'invited'), а `deal_members.role_id` — гранулярная роль внутри сделки.
- **Fail-closed (Запрещено по умолчанию)**: Всё, что не разрешено явно, должно блокироваться базой данных (выдавать пустой результат при чтении или ошибку RLS при записи).

---

## 3. Вспомогательная функция (Helper Function) — Контракт и Ограничения

Для предотвращения проблемы рекурсивного RLS (когда политика для `deal_members` пытается запросить саму себя), а также для унификации правил, внедряется функция-контракт `has_deal_role`.

**Важнейшие ограничения для EXECUTOR:**
- Функция **ОБЯЗАНА** иметь маркировку `SECURITY DEFINER` и установку `SET search_path = public`, чтобы обходить RLS внутри своего тела, так как она вызывается в момент проверки политик.
- Функция должна иметь скалярный список ролей (или использовать массив строго определенных типов). Если парсинг массивов вызывает сложности при написании SQL, EXECUTOR имеет право (но не обязан) переписать контракт функции для приема обычных строковых флагов (например, `p_role1 TEXT DEFAULT NULL, p_role2 TEXT DEFAULT NULL...`).
- Контракт (ожидаемое поведение):
  - Принимает: `p_deal_id UUID` и один или несколько текстовых идентификаторов ролей.
  - Возвращает: `BOOLEAN`. Возвращает `TRUE`, если `auth.uid()` присутствует в `deal_members` для `p_deal_id` и его роль (`deal_roles.name`) совпадает с одной из переданных.
  - Если список ролей пуст/NULL — возвращает `TRUE` при наличии *любой* роли в сделке (допуск для всех участников).
- **Альтернатива (если функция вызывает лаги или ошибки Supabase)**: Отказаться от функции и встроить прямые `EXISTS (SELECT 1 FROM deal_members ...)` в каждую политику.

## 4. Якорные пути и политики по таблицам (Table-by-Table Scope)

Таблицы ниже определены относительно их якорной привязки к сделке (deal anchor path). Настройка RLS **ОБЯЗАНА** использовать именно этот точный путь, а не теоретический.

### A. Базовые объекты сделки (Прямой `deal_id`)
*Включает таблицы:* `deals`, `buyers`, `pipeline_stages`, `documents`.
- **Якорь**:
  - для `deals` — прямой `id`
  - для `buyers`, `pipeline_stages`, `documents` — прямая колонка `deal_id`
- **Право ЧТЕНИЯ (SELECT)**: Разрешено всем участникам сделки (любая роль в `deal_members`).
- **Право ЗАПИСИ (INSERT/UPDATE/DELETE)**:
  - `deals`, `buyers`, `pipeline_stages`: **Только `lead_advisor`**. (Роль `advisor` не имеет прав модифицировать воронку или параметры сделки).
  - `documents`: `lead_advisor` и `advisor` (обе могут загружать и удалять документы). `viewer` не может.

### B. Связанные объекты ядра (Косвенный `deal_id`)
*Включает таблицы:* `buyer_pipeline_states`, `communications`.
- **Якорь**: Через колонку `buyer_id` -> `buyers.deal_id`. (В политике требуется `JOIN` или подзапрос к `buyers`).
- **Траектория RLS**: `EXISTS (SELECT 1 FROM buyers WHERE buyers.id = table.buyer_id AND has_deal_role(buyers.deal_id, ...))`
- **Право ЧТЕНИЯ (SELECT)**:
  - `buyer_pipeline_states`: Все участники сделки (`lead_advisor`, `advisor`, `viewer`).
  - `communications`: **Только `lead_advisor` и `advisor`**. Для `viewer` доступ полностью закрыт (возвращает 0 строк).
- **Право ЗАПИСИ (INSERT/UPDATE/DELETE)**:
  - `buyer_pipeline_states`: **Только `lead_advisor`**.
  - `communications`: **Только `lead_advisor` и `advisor`**. Для `viewer` доступ полностью блокируется политиками RLS на уровне БД.

### C. Операционные объекты (Прямой `deal_id`)
*Включает таблицы:* `tasks`.
- **Якорь**: Прямая колонка `deal_id`.
- **Право ЧТЕНИЯ/ЗАПИСИ (ALL)**: Разрешено только `lead_advisor` и `advisor`. Для `viewer` доступ закрыт на уровне базы данных.

### D. ИИ Очередь (Прямой `deal_id`)
*Включает таблицы:* `ai_jobs`, `ai_outputs`.
- **Якорь**: Прямая колонка `deal_id`.
- **Право ЧТЕНИЯ/ЗАПИСИ (ALL)**: Разрешено **только `lead_advisor`**. Все остальные роли получают отказ.

## 5. Доверенные системные потоки (Trusted Server Paths)

### A. Членство (Таблица `deal_members`)
**Критическое ограничение**: Обычные пользователи-клиенты (даже `lead_advisor`) **НЕ ДОЛЖНЫ** напрямую редактировать `deal_members` через клиентские `INSERT / UPDATE / DELETE` запросы RLS.
- **R/W Модель**: Управление членством, как и принятие приглашений (через `src/actions/invite.ts`), выполняется строго через защищенный серверный `createAdminClient()`.
- **Политика RLS (`deal_members`)**: Должна быть оставлена только на **ЧТЕНИЕ (SELECT)**: `has_deal_role(deal_id, NULL)` (любой участник сделки может прочитать список участников). Никакие `INSERT`, `UPDATE` или `DELETE` в публичных политиках RLS для `deal_members` не создаются.

### B. Приглашения (Таблица `deal_invites`)
Приглашения остаются защищенными серверными скриптами, RLS расширять запрещено.
- **Политика RLS**: Текущая политика `invited_by = auth.uid()` остается каноничной. Никто не может "посмотреть" чужие инвайты через клиентский API. Инвайты валидируются и принимаются сервером по токену в обход RLS.

### C. Пользователи и Организации (Residual Exposure)
Перенос RLS на сделки не избавляет полностью от необходимости показывать участникам профили друг друга. Данная проблема профилей (`users`) классифицирована как **принятый временный остаточный риск (accepted temporary residual risk)**, выходящий за рамки Этапа 8.
- **Действие**: Изменять существующие публичные (public) политики для `users` и `organizations` **НЕ НУЖНО**. Главная цель Этапа 8 — закрыть права записи/чтения именно к бизнес-сущностям сделки (`pipeline`, `documents`, `tasks`), которые сейчас открыты для всего `organization_id`. Скрытие профилей продавцов друг от друга или от покупателей в будущем — цель будущих этапов.

## 6. Что именно НЕ должен делать Stage 8 EXECUTOR (Красные линии)

Выполняющему SQL-команды агенту **БЕЗУСЛОВНО ЗАПРЕЩЕНО**:
1.  Открывать прямые публичные RLS права на запись (INSERT/UPDATE/DELETE) для таблицы `deal_members`.
2.  Смягчать видимость (RLS) таблицы `deal_invites`.
3.  Изменять/удалять существующее серверное присвоение `organization_id` в логике приема инвайтов `src/actions/invite.ts` (Стейдж 7/мост должны остаться функциональны, скрипты тестирования не должны сломаться).
4.  Создавать нереплицируемые или недокументированные зависимости в БД.
5.  Использовать в миграциях деструктивные команды наподобие `DROP TABLE`.
6.  Решать проблему видимости профилей (Profile/Org visibility) — не нужно править RLS таблиц `users` / `organizations`. Оставьте их как есть.
7.  Придумывать поля таблиц или якорные ключи, которых не существует (например, `components`, 'deals.user_id' и т.п.).

## 7. Стратегия Развертывания и Регрессии

- **Шаг 1**: Изучить существующие политики RLS (через выборку из `pg_policies`).
- **Шаг 2**: Написать скрипт миграции, который сначала создает хелпер-функцию.
- **Шаг 3**: Скрипт миграции делает `DROP POLICY` только для бизнес-таблиц сделки (Таблицы в секции 4). Таблицы профилей и административные обходит стороной.
- **Шаг 4**: Скрипт создает новые `CREATE POLICY` на основе хелпера и якорей из секции 4.
- **Шаг 5 Проверка (Regression Strategy)**: После запуска EXECUTOR должен убедиться, что:
    - Политика `deal_members` разрешает только `SELECT`.
    - Политика `ai_jobs` не разрешает доступ `viewer` и `advisor`.
    - RLS-loop/рекурсий на сервере нет.
    - Старый `organization_id` больше не упоминается в новых политиках для `deals` и CRM-сущностей.
- **Stop Conditions для Stage 8 EXECUTOR**:
  - Если структура таблиц БД (поля/связи) расходится с описанными здесь Якорями — немедленная остановка и откат.
  - Ошибка синтаксиса или конфликты прав при написании SQL (сделать откат `BEGIN...ROLLBACK` или `psql -1`).
