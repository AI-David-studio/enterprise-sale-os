# Этап 9: Базовая CRM-функционализация (Deals, Buyers, Pipeline)

## 1. Название этапа
Stage 9: Core CRM Functionalization (Базовые мутации: Сделка, Покупатели, Воронка).

## 2. Цель этапа
Создать первые реальные механизмы изменения данных (мутации) на защищенной стороне продавца. Заменить существующие UI-заглушки (кнопки "Добавить", "Сохранить") на работающие Server Actions. Этап строго сфокусирован на **базовом скелете**: редактировании свойств сделки, создании покупателей (без их изменения) и переводе покупателей по этапам воронки.

## 3. Почему этот этап следует за Этапом 8
На Этапе 8 мы внедрили строгий контроль доступа (RLS), привязанный к членству в сделке (`deal_members`). Теперь база данных безопасно блокирует неавторизованную запись (Fail-closed).
Текущий интерфейс продавца состоит из маршрутов (`/deal`, `/buyers`, `/dashboard`), которые успешно читают данные под защитой RLS, но все кнопки изменения данных отключены (`disabled` или ведут в никуда).
Перед тем как делать операционные фичи (документы, задачи, ИИ), нам нужен работающий "скелет" сделки: возможность переименовать саму сделку, добавить покупателей и двигать их по этапам. Без этого скелета все остальные таблицы просто не к чему привязывать в UI.

## 4. Аудит текущего состояния (Current Surface Audit)
- **`/dashboard`**: Сводка только для чтения.
- **`/deal` (Настройки сделки)**: Форма содержит реальные данные из БД, но кнопка "Сохранить изменения (Заглушка)" отключена. Нет Action'а для `UPDATE deals`.
- **`/buyers` (Список)**: Отображает реальных покупателей. Кнопка "+ Добавить покупателя" отключена. Нет модалки/формы создания.
- **`/buyers/[id]` (Карточка)**: Показывает текущий этап воронки (`buyer_pipeline_states`), но нет интерфейса, чтобы перевести покупателя на следующий этап.
- **`/tasks`, `/communications`, `/ai-review`**: Полные заглушки без мутаций (имеют кнопки `disabled`).

## 5. Обоснование выбора Этапа 9
Мы выбираем **Core CRM (Deals + Buyers + Pipeline)** по следующим причинам:
1. **Зависимость маршрутинга**: Нельзя ставить задачи по покупателю или генерировать письма, если покупателя нельзя создать через UI.
2. **Изоляция риска**: Логика работы со сделкой и покупателями прямолинейна (CRUD) и идеально подходит для проверки новой модели прав RLS (через `lead_advisor`), прежде чем мы перейдем к сложным многоступенчатым фичам (например, генерации AI черновиков с сохранением).
3. **Ценность (MVP Value)**: Позволяет Lead Advisor'у реально начать вести реестр покупателей в рамках сделки из UI.

## 6. Сущности в фокусе (In-Scope Entities)
- `deals` (только `UPDATE` существующих полей `name`, `description`, `target_industry`).
- `buyers` (только `INSERT` — создание нового покупателя. `UPDATE` покупателя вне рамок этапа).
- `buyer_pipeline_states` (`INSERT` при создании покупателя, `UPDATE` при переводе по воронке).
- `pipeline_stages` (только `SELECT` для вывода списка этапов).

## 7. Включено в охват (In-Scope Pages / Actions)
1. **Настройки сделки (`/deal`)**:
   - Создать Server Action для обновления `name`, `description`, `target_industry`.
   - "Оживить" форму, включить валидацию Zod, кнопки состояния "Сохраняется...".
2. **Список покупателей (`/buyers`)**:
   - Создать Server Action `createBuyer(dealId, data)`.
   - Реализовать простой UI (компонент-модалку) для создания покупателя. В форме только: `name`, `industry`, `website`, `description`.
   - **Строгое правило пайплайна**: Создание покупателя ДОЛЖНО атомарно создавать его начальный стейт в `buyer_pipeline_states`, используя этап из `pipeline_stages` с минимальным `sort_order` для данной сделки. Если этапов для сделки нет, бросать понятную ошибку.
3. **Карточка покупателя и Воронка (`/buyers/[id]`)**:
   - Создать Server Action `updateBuyerPipelineStage(buyerId, newStageId)`.
   - Добавить в UI селектор или кнопки переключения этапов воронки (извлекая доступные этапы из `pipeline_stages`).

## 8. За границами охвата (Out-of-Scope)
- **Не менять RLS**. Политики Этапа 8 зафиксированы. Разрешено только полагаться на них (если попытается изменить `viewer`, БД сама откажет).
- **Мутации Tasks, Communications, AI, Documents**: Остаются в статусе "только чтение" или `disabled`. Не трогать эти разделы.
- **Удаление (DELETE) и Редактирование (UPDATE) покупателей**: Вне рамок этапа для снижения риска слома. Только создание и перемещение стейта.
- **Сложная кастомизация воронки**: Lead Advisor не должен создавать новые `pipeline_stages` на этом этапе.
- **Никаких Data Migrations**: Этап 9 использует СУЩЕСТВУЮЩУЮ структуру БД и текущие данные из Seed. Запрещено модифицировать `seed.sql`, создавать миграции или придумывать fallback-скрипты. Если данных нет, возвращаем UI ошибку "Нужна настройка этапов".
- **Управление доступом**: Авторизация и инвайты (Стейдж 7) остаются как есть, новые роли не добавляются.

## 10. Ожидаемое поведение (Target Behavior)
После завершения Этапа 9 `lead_advisor` сможет:
1. Зайти в Настройки, изменить название проекта и описание, нажать Сохранить — данные обновятся в БД.
2. Зайти в Покупатели, нажать Добавить, заполнить форму — новый покупатель появится в списке.
3. Открыть карточку покупателя, выбрать из дропдауна новый этап воронки (например, "NDA Signed") — счетчик/статус обновится.
4. При попытке совершить любую из этих операций пользователем-`viewer`, сервер должен выбросить ошибку прав доступа (но лучше вообще скрыть кнопку в интерфейсе).

## 11. Границы выполнения (Execution Boundaries)
- **Только Server Actions + React Components**: Никаких новых миграций БД (разве что `seed.sql` для этапов, если нужно).
- **Изоляция в папках**: `src/actions/` (новые файлы `deal.ts`, `buyer.ts`), `src/app/(protected)/deal/`, `src/app/(protected)/buyers/`.
- **Без Client-Side Data Fetching**: Использовать текущий паттерн RSC (React Server Components) и Server Actions. Состояние загрузки (pending) реализовать через `useFormStatus` или `useTransition`.

## 12. Критерии приемки (Acceptance Criteria)
1. Реализован Server Action для обновления полей `name`, `description`, `target_industry` сделки. В Action встроена проверка `getUserDealRole === 'lead_advisor'`.
2. Реализован Server Action для создания покупателя (`name`, `industry`, `website`, `description`). При создании покупателя ему немедленно присваивается первый этап воронки (наименьший `sort_order` из `pipeline_stages` сделки).
3. Реализован Server Action для перевода покупателя на другой этап воронки через интерфейс `/buyers/[id]`.
4. Все три новых Server Actions явно проверяют права `lead_advisor` перед обращением к БД, возвращая ошибку авторизации, а не полагаясь только на RLS как на первое кольцо защиты. RLS действует как fail-closed backstop.
5. Неавторизованные попытки (или вызовы для ролей `viewer/advisor`) приводят к ошибке и не изменяют БД.
6. Файлы Stages 1-8 не сломаны (особенно `middleware.ts` и флоу инвайтов).

## 13. Условия остановки (Stop Conditions)
- Остановка, если выясняется, что в БД отсутствует привязка обязательных параметров (например, невозможно связать `buyer_pipeline_states` с существующими `pipeline_stages`).
- Остановка, если реализация `PipelineStageSelector` требует сложного стейт-менеджмента клиентской стороны (Redux/Zustand) — MVP требует простых нативных форм React 19 / Next.js App Router Server Actions.
